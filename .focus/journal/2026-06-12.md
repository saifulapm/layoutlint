# 2026-06-12 — `layoutlint render` built end-to-end (feat/render, 9 commits)

Built the "agent eyes" feature on top of the geometry engine: deterministic
SVG/PNG rendering without a browser, LARGE plan, all 8 tasks executed and
evaluator-PASSed.

- Visual resolver (core/visual.ts): colors/radius/text-align from the classes
  the layout resolver ignores; CSS color inheritance; layout Style + goldens
  untouched (geometry stayed 297/297 through every task).
- Palette: extracted Chrome's exact sRGB bytes for all 244 Tailwind v4 colors
  via canvas getImageData against the vendored TW build (no oklch math).
- Painter (core/paint.ts): nested rounded rects for bg/border, overflow
  clipPaths, text as glyph-outline paths from the same HarfBuzz shaping that
  measured it (breakLines shared by construction).
- render() + CLI `render` + MCP `render_layout` (PNG image content block);
  PNG via @resvg/resvg-js (new runtime dep, externalized in bundle).
- Validation: screenshot oracle (pinned Chromium, srgb profile) + pixelmatch
  gate in CI — 42/42 cases ≤5% mismatched pixels, worst 3.23% (Bangla glyph
  AA), median ~0.9%; screenshots regenerate byte-identically (evaluator
  verified). Visual spot-checks: engine vs Chrome indistinguishable.
- Bugs caught by looking at renders: glyph scale rounded to 2dp (46% glyph
  inflation at 14px); text leaves not painting their own bg/border after
  parse-collapse; CBDT emoji glyphs have no outlines (null guard + bitmap
  fallback).
- Deviations recorded in plan: srcless <img> paints nothing (browser parity);
  threshold stays 5% until Linux CI baselines; inner-span colors share the
  existing parse-collapse envelope.

## Retro: layoutlint render (2026-06-12)
- What went well: the parallel-resolver design meant zero risk to goldens —
  geometry stayed green through all 8 tasks; rendering PNGs and *looking* at
  them caught two real bugs the tests missed.
- What went poorly: two plan details didn't survive contact (img placeholder
  vs browser parity; scale precision) — both caught by validation, but the
  placeholder one should have been caught at plan time by asking "what does
  Chrome paint for a srcless img?"
- Change for next time: for any paint/visual feature, write the oracle
  comparison FIRST (task 7 before task 5) so every painter decision is
  validated the moment it's made, not at the end.
